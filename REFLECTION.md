# REFLECTION.md — CodeCheck 项目反思报告

> 本文由学生本人撰写，AI 辅助润色与结构整理。基于 Superpowers 方法论完成 CodeCheck 编码智能体 Harness 项目后的系统性反思。

---

## 一、Superpowers 技能：哪些真正有用，哪些流于形式

### 1.1 发挥最大作用的技能

**Brainstorming（7 轮递进提问）** 是整个项目最重要的技能。它从宏观定位（"反馈闭环作为深度维度"）到微观机制（"B 模式修复流程"），通过 24 个问题逐层收敛，帮我建立了完整的项目框架。如果没有这个结构化过程，我大概率会直接跳到"写一个能审查代码的脚本"，而不会思考 harness 的六个维度应该如何设计、反馈信号应该如何回灌。

**Test-Driven Development** 是第二重要的技能。在 AI 生成代码的协作模式下，TDD 的最大价值不是"保证代码正确"（传统意义上的），而是**让 AI 的输出可验证**。AGENT_LOG 中记录的多个案例——T2 的 keyword 字面匹配、T3 的 fixture 文件名、T6 的 2 个 rule 产生 2 个 issue——都是测试先于实现暴露的问题。如果没有 TDD，这些问题会在集成阶段才被发现，修复成本会高得多。

**Using Git Worktrees** 提供了干净的隔离环境。但对于本项目（单人开发、15 个 task、总代码量约 3000 行），worktree 的性价比不算高——创建和切换 worktree 的开销，有时候比 task 本身的开发时间还长。不过我能理解它的设计意图：在多人协作或大型项目中，worktree 隔离是必要的。

### 1.2 "形式大于实质"的技能

坦白说，**Requesting Code Review** 和 **Finishing a Development Branch** 这两个技能在本项目中几乎没有发挥作用。原因很简单：我是唯一的开发者，没有"同事"可以请求 review，也不存在"多个 PR 需要决定合并/丢弃"的场景。这两个技能更像是为团队协作设计的，在单人项目中它们的存在感很弱。

但这不意味着它们没用——如果这是一个团队项目，我会需要它们。只是在本项目的语境下，它们确实走了形式。

### 1.3 如果去掉某个技能的限制

如果去掉 TDD 的强制要求，我怀疑 T11-T15（CLI、Hook、Docker、CI、README）的代码质量会下降——这些"胶水层"任务很容易被当成"不重要"而跳过测试。但反过来，T11-T15 也确实不需要像 T10（反馈闭环）那样严格的红-绿-重构循环。一个更务实的做法是：**核心机制（T2-T10）严格 TDD，集成层（T11-T15）允许实现后补测试**。

## 二、TDD 在 AI 协作下的角色：阻碍还是放大器？

### 2.1 整体判断：放大器

TDD 在 AI 协作下是**放大器**，而非阻碍。原因在于：AI 生成的代码有一个特点——"看起来对"但"细节容易错"。TDD 的作用就是把"细节错误"提前暴露出来，避免它们在集成阶段爆炸。

从 AGENT_LOG 的统计数据看，15 个 task 中有 8 个 task 出现了"人工干预修复测试"的情况。这些修复大部分是测试用例和实现之间的 mismatch——比如 T2 的 `test_keyword_match` 中测试内容不包含关键词、T6 的 `test_llm_finds_issues` 中 LLM 对每个 rule 独立调用导致 issue 数量不符预期。没有 TDD，这些问题会在 Agent 主循环集成时以"LLM 输出不符合预期"的形式出现，届时定位问题的难度会大得多。

### 2.2 哪里 TDD 价值不大

T11-T15 这 5 个 task 是"胶水层"——它们不包含核心逻辑，主要是把已有的模块连线、生成配置、写文档。对这些 task 严格执行 TDD，性价比不高。比如 T15（README 文档）根本没有"测试"可言；T13（Dockerfile）的测试本质上是 `docker build` 能否成功，这属于 CI 层面的验证而非单元测试。

如果重来，我会把 TDD 的要求分为两级：**核心机制（T1-T10）红-绿-重构**，**集成与分发（T11-T15）实现后统一测试**。

### 2.3 最有信心的测试

268 个测试中，我最信任的是 T10（反馈闭环）的 22 个测试。它们完整覆盖了"修复→验证→回滚→重试→收敛"的整个链路，使用 mock LLM 注入不同场景（第 1 轮失败第 2 轮成功、3 轮全部失败、修复应用失败等），而且所有测试都不依赖网络或真实 LLM。这恰好符合项目要求的判定标准：**"移除 LLM 后，机制还能用单测验证吗？"**——T10 的测试完美地回答了"能"。

## 三、Subagent 工作流与 Task 颗粒度

### 3.1 AI 偏离的发现方式

在开发过程中，AI 确实出现过偏离——不是"生成了完全不相关的代码"，而是"当开发效率下降时，细节开始出错"。我主要通过**直接阅读代码**来发现这些问题，而非等测试挂掉。这种方式效率更高，因为测试挂了之后还需要回溯代码定位问题，而直接读代码可以当场判断"这个逻辑不对"。

### 3.2 最优 Task 颗粒度

PLAN 中 15 个 task 的粒度差异很大。从实际体验看：

- **T1-T8（Phase 1-2）的颗粒度最好**。每个 task 聚焦一个独立模块（LLM 抽象、配置、凭据、工具、规则、护栏、记忆），依赖清晰，可并行开发。时间跨度约 10-15 分钟。
- **T11-T15（Phase 3-4）拆得过细**。T11（CLI）和 T12（Hook）本质上是"把已有模块连线到 CLI"，应该合并；T13（Docker）、T14（CI）、T15（README）也应该合并，因为它们都是"分发与文档"，没有代码依赖。

如果重做，我会把 15 个 task 合并为 12 个：T1-T10 保持不变，T11+T12 合并为"CLI 与 Hook 集成"，T13+T14+T15 合并为"分发、CI 与文档"。

### 3.3 最优时间跨度

我认为一个 task 的最优时间跨度是 **10-15 分钟**。这个时间足够 AI 完成一个有意义的代码单元（一个模块 + 其测试），但又不会长到上下文窗口膨胀、AI 开始偏离的程度。T10（反馈闭环）是唯一一个超过这个时间跨度的 task，因为它的逻辑确实复杂——但即使如此，如果能拆成"备份机制"、"验证器"、"主循环"三个子 task，可能会更可控。

## 四、SPEC/PLAN 质量与实现质量的关系

### 4.1 最有效的 Prompt 要素

PLAN.md 中每个 task 包含五个要素：**目标、涉及文件、预期实现要点、验证步骤、依赖关系**。其中对 AI 生成质量影响最大的是三个：

1. **目标**：一句话说清楚"这个 task 要产出什么"
2. **伪代码**（预期实现要点中的代码片段）：给 AI 一个"正确答案的形状"
3. **验证步骤**：明确告诉 AI "做到什么程度才算完成"

我给自己定下的两条 prompt 铁律是：**"永远先读文件再修改"** 和 **"遇到不确定之处暂停询问"**。前一条避免了 AI 在不知道文件内容的情况下盲目修改，后一条避免了 AI 凭猜测继续导致错误积累。

### 4.2 规约不清导致偏离的具体案例

最典型的案例是 T11 的 `.codecheck/rules.yaml` 的 YAML 转义问题。

SPEC 中定义了规则 `sql-string-concat` 的正则 pattern 为 `('f'|['\"])\\s*(SELECT|INSERT|UPDATE|DELETE)\\s`。这个 pattern 在 Python 正则中是正确的，但把它写入 YAML 双引号字符串时，`\s` 会被 YAML 解析器视为非法转义序列（YAML 双引号字符串中 `\s` 不是合法转义）。

PLAN 中只写了"内置 5 条确定性规则（仅 YAML 定义，T6 才写匹配逻辑）"，但没有说明 YAML 的转义规则。AI 在 T1 生成 rules.yaml 时使用了双引号包裹 pattern，导致 `\s` 被 YAML 解析器拒绝。这个问题直到 T11 实现 review 命令时，因为 rule engine 加载 rules.yaml 失败才暴露出来。

修复方案是改用 YAML 块标量语法 `|-`，这样 `\s` 会被原样保留。这个问题的根因是：**SPEC 定义了"规则的正则表达式"，但没有定义"正则表达式在 YAML 中的表示方式"**——这是一个跨模块的接口约定缺失。

## 五、凭据安全与分发：工程要求的真正价值

### 5.1 凭据安全

在实现 T4 之前，我处理 API Key 的方式很"原始"：创建一个 `.env` 文件，写一行 `API_KEY=sk-xxx`，然后在 `.gitignore` 里加上 `.env`。这确实能防止 key 被提交到 Git，但有几个问题：key 以明文存储在磁盘上（任何能读取文件系统的进程都能看到）、没有加密保护、没有引导用户录入的流程。

T4 的强制性要求迫使我思考了这些问题。最终实现的 Fernet 加密 + PBKDF2 密钥派生 + 600 文件权限，虽然对于一个小型项目来说有些"过度工程"，但它让我第一次认真思考了凭据的威胁模型：**攻击者能做什么？我们能防什么？防不住的边界在哪里？**

### 5.2 分发

分发这条要求对我的触动更大。坦白说，我原来做的项目在"教别人用"上基本没有投入——代码写完、README 写个大概、push 到 GitHub，就结束了。别人怎么跑起来、遇到问题怎么办，我没有认真想过。

Docker 分发（T13）迫使我从"用户视角"思考：一个全新用户从零开始运行 CodeCheck，他需要什么？Docker 已安装、知道挂载卷、知道如何配置 API key——这些对于熟悉 Docker 的开发者来说是常识，但对于不熟悉的人就是障碍。README 中的"Key 安全配置"表格（加密存储 vs 环境变量 vs 硬编码）就是在这种思考下产生的。

### 5.3 哪个触动更大

分发。因为凭据安全是"防御性"的——防止坏事发生；分发是"进攻性"的——让好事发生（让别人真的能用起来）。后者的思维方式对我而言更新鲜。

## 六、如果重做，我会改变什么

### 6.1 Task 合并

如第三节所述，T11+T12 合并，T13+T14+T15 合并。减少 worktree 的创建和切换开销。

### 6.2 测试策略分级

核心机制（T1-T10）严格 TDD，集成层（T11-T15）实现后统一测试。这样既保证了核心逻辑的可验证性，又不会在胶水代码上浪费 TDD 的仪式感。

### 6.3 SPEC 和 PLAN 花更长时间

冷启动验证（用另一个 agent 仅凭 SPEC+PLAN 尝试实现）是 Superpowers 流程中我最认可的一步。另一个 agent 在不知道我的"隐性上下文"的情况下，暴露了 SPEC 中的多个模糊之处（如 YAML 转义、工具注册顺序、Agent 主循环的停机条件）。这些反馈让我在正式实现前修补了 SPEC 和 PLAN 的很多细节。如果重做，我会在 SPEC 和 PLAN 阶段花更多时间，让冷启动验证更充分——因为 **SPEC 的质量直接决定了后续所有 task 的实现质量**。

### 6.4 技术选型

我选择 Python 的主要原因是"我只熟悉 Python"。AI 在 brainstorming 时没有追问"是否考虑过 Go/Rust"，但即使问了，我大概率还是会选 Python——因为本项目的目的不是追求极致性能，而是验证 harness 的工程设计。Python 的生态（Click、cryptography、pytest、ruff）对于快速原型开发非常友好。

## 七、对 Superpowers 方法论的批判

### 7.1 它的核心假设

Superpowers 的核心理念是"Spec-Driven, Subagent-Built, Human-Owned"。它假设：

1. **只要 SPEC 足够清晰，subagent 就能独立完成任务**
2. **TDD 的"红-绿-重构"循环在 AI 协作下仍然有效且必要**
3. **Git worktree 隔离 + 独立 PR 是最优的并行开发方式**
4. **人的角色是"审查者"和"决策者"，而非"编码者"**

### 7.2 哪些假设成立，哪些打了折扣

**假设 1——打了折扣。** SPEC 的清晰度确实是 subagent 成功的关键，但"清晰"有两个层面：(a) 规约本身的完整性和准确性，(b) subagent 的上下文窗口能否装下足够的规约信息。即使 SPEC 写得很好，如果 task 拆得太细（比如 T11-T15），subagent 可能因为缺少上下文而做出不一致的设计决策。所以这个假设成立的前提是：**SPEC 清晰 + task 粒度合理 + 模型上下文窗口够大**。

**假设 2——成立。** TDD 在 AI 协作下甚至是更重要的——因为 AI 生成的代码"看起来正确"的概率远高于"真正正确"的概率，TDD 是唯一的客观验证手段。

**假设 3——部分成立。** Worktree 隔离在多人协作或大型项目中非常有效，但在单人小型项目中性价比不高。不过这更多是项目规模的问题，而非方法论本身的问题。

**假设 4——成立。** 这是我对 Superpowers 最认同的一点。在整个项目中，我几乎没有手写任何代码——所有代码都由 AI 生成。但我做了大量的决策：选什么技术栈、做到什么深度、哪里的 AI 输出需要修正、测试怎么设计才真正有意义。这些决策的质量，而不是代码量，决定了项目的最终质量。

### 7.3 这套方法论最适合什么项目

Superpowers 最适合**需求一开始就固定的项目**。因为 PLAN 在开发开始前就定死了——task 的依赖关系、验证步骤、预期实现要点都是提前写好的。如果需求频繁变化，PLAN 的维护成本会很高。

它也最适合**模块边界清晰的项目**。CodeCheck 的六个维度（决策、工具、记忆、治理、反馈、配置）天然就是六个独立模块，这让 task 拆分和并行开发变得非常自然。如果是一个单体应用，模块之间的耦合会让 worktree 隔离变得困难。

### 7.4 如果加一条原则

我会加：**"SPEC 和 PLAN 花的时间尽量长一些"**。因为一旦进入实现阶段，SPEC 和 PLAN 中的任何模糊之处都会在代码中放大。冷启动验证是发现这些模糊之处的最好机制，但它需要 SPEC 和 PLAN 已经达到一定的完整度。多花时间在规约上，是对后续所有实现时间的投资。

## 八、结语：一个工程师在 AI 时代的价值

这门课的核心命题是：**"当 LLM 能完成大部分编码工作时，一个工程师的真正价值在哪里？"**

经过这个项目，我的答案是：**在 harness 这一层**。LLM 负责"决定下一步做什么"，但决定做得对不对、动作是否安全、失败了怎么恢复、上下文如何组织——这些是 harness 的职责，也是工程师的职责。CodeCheck 项目中的 268 个测试，没有一个依赖真实 LLM——它们验证的全部是 harness 的确定性逻辑。这说明：**真正可测试、可依赖的工程，不在 LLM 的"智能"里，而在 LLM 之外的那层代码里。**

Superpowers 给我的是"流程脚手架"，它守住了 TDD、review、planning 这些在 AI 协作中容易松懈的纪律。但它不能替我做决定——"做什么"和"做对了吗"这两个问题，始终需要人来回答。这大概就是这门课想教给我的东西。
