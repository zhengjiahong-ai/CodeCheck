## 一、Brainstorming 关键节点

### 1.1 智能体追问的关键问题

在 brainstorming 过程中，智能体共提出了 **24 个问题**，分 7 轮递进。以下是对设计产生实质性影响的关键节点：

**第一轮：核心定位（问题 1-2）**
- 智能体追问："你要选哪个维度深入？"
- 我的选择：**反馈闭环** 作为深度维度，**自动代码审查** 作为应用场景
- 为什么：反馈闭环的每个环节都可以编码实现，最符合 harness 的"机制必须是代码"要求

**第二轮：场景细化（问题 3-5）**
- 智能体追问："反馈信号是什么？"
- 我的选择：检测代码 pattern → 与已知 bug 数据库/best practice 规则匹配（混合模式）
- 智能体追问："护栏是什么？"
- 我的选择：所有列出的操作（修改文件、执行 shell、commit、安装依赖、删除文件）都需要受限制
- 智能体追问："技术栈？"
- 我的选择：DeepSeek + Python

**第三轮：机制设计（问题 6-9）**
- 智能体追问："规则体系如何设计？"
- 我的选择：混合模式（确定性正则匹配 + LLM 语义理解）
- 智能体追问："闭环的修正步骤？"
- 我的选择：B 模式——自动修复 → 自动应用 → 运行测试 → 失败则回滚 + 重试
- 智能体追问："记忆维度？"
- 我的选择：审查历史、项目规则、误报记录都需要
- 智能体追问："深度做到哪一层？"
- 我的选择：深度（多轮修复-验证循环 + 失败分类 + 收敛判断）

**第四轮：主循环与工具（问题 10-13）**
- 智能体追问："触发方式？"
- 我的选择：CLI + git hook
- 智能体追问："Agent 需要哪些工具？"
- 我的选择：必须和建议有的都保留
- 智能体追问："回滚机制？"
- 我的选择：文件级备份恢复
- 智能体追问："增量审查？"
- 我的选择：支持，通过 git diff

**第五轮：架构细节（问题 14-17）**
- 智能体追问："主循环长这样？"
- 我的确认：没问题
- 智能体追问："记忆存储方案？"
- 我的选择：SQLite + YAML + ChromaDB 组合
- 智能体追问："护栏触发条件？"
- 我的确认：权限矩阵合适
- 智能体追问："项目名称？"
- 我的选择：CodeCheck

**第六轮：非功能需求（问题 18-21）**

- 我的选择：只标记"需人工介入"并输出已尝试的修复历史

- 我的选择：Docker

**第七轮：收尾确认（问题 22-24）**

- 我的确认：MVP 为 US-1 到 US-3，US-4 到 US-7 为后续迭代
- 智能体追问："配置文件格式？"
- 我的选择：`.codecheck.yaml` 项目根目录
- 智能体追问："Mock LLM 策略？"

### 1.2 AI 提出而我采纳的建议

1. **分轮递进式提问**：AI 将 brainstorming 分为 7 轮，每轮 3-4 个问题，逐步收敛。这种结构化方式让我不需要一次性回答所有问题，降低了决策负担。

2. **主循环流程图**：AI 在问题 14 中给出了一个伪代码式的主循环流程，让我能直观看到整个系统的运行逻辑，并在此基础上确认/调整。

3. **权限矩阵表格**：AI 在问题 16 中用表格列出每个操作的权限级别（自动/需确认/禁止），比文字描述清晰得多，我直接确认了。

4. **规则 YAML 格式设计**：AI 在问题 19 中给出了一个具体的规则文件格式示例，包含 `id`、`severity`、`type`、`pattern`/`prompt` 等字段，这个设计我全盘采纳。

5. **深度等级表**：AI 在问题 9 中用表格列出了反馈闭环从"基础"到"极致"的四个等级，帮助我快速定位目标深度。

6. **数据模型设计**：AI 在 SPEC 中主动设计了完整的 SQLite 表结构和 ChromaDB 集合，超出了我的预期。

### 1.3 我推翻或修正的 AI 建议

1. **问题 3 的选项 C vs A/B**：AI 给出了三个反馈信号思路（A: 测试驱动、B: 人工评判 review 质量、C: bug 数据库/best practice 匹配），我选了 C。A 和 B 更偏向"写代码"场景，而 C 更贴合"代码审查"的定位。

2. **问题 7 的修复流程**：AI 给出了三个选项（A: 生成建议用户审阅、B: 自动修复→测试→回滚重试、C: 双 agent 复核），我选了 B。这是最核心的决策——B 模式让反馈闭环名副其实，而 A 模式本质上只是"生成建议"。

3. **问题 12 的回滚粒度**：AI 给出了三种选项（A: git reset --hard、B: 文件级备份恢复、C: 事务式 checkpoint），我选了 B。A 太粗暴（可能丢失用户自己的未提交修改），C 太复杂。

4. 其他问题我基本确认了 AI 的设计，没有重大修正。

## 三、反思

### 3.1 Brainstorming 技能做得好的地方

1. **结构化递进**：7 轮、24 个问题，从宏观定位到微观机制，逐层收敛。这种递进式提问避免了信息过载，也确保了每个决策都有足够的上下文。

2. **在每个关键决策点给出选项**：AI 几乎从不问开放式的"你想怎么做"，而是给出 2-4 个具体选项让我选择。这大大降低了决策难度——识别"哪个选项适合我"比"从零设计"容易得多。

3. **用表格和伪代码可视化复杂设计**：主循环伪代码、权限矩阵、深度等级表、规则 YAML 格式——这些可视化的中间产物让我能快速理解设计意图并确认。

4. **最后输出设计摘要**：在 brainstorming 结束时做了一个完整的汇总，让我在进入 SPEC 写作前有一个全局确认的机会。

5. **安全提醒**：在开始 brainstorming 前，AI 发现并提醒了 settings.json 中的 API Key 泄露问题，这体现了工程安全意识。

### 3.2 Brainstorming 让我不满意的地方

1. **缺少对"为什么不选其他方向"的追问**：当我选择"反馈闭环"时，AI 没有追问"为什么不选治理/记忆/编排"——如果追问了，可能会帮助我更好地论证自己的选择，或者发现我遗漏的考量。

2. **缺少对边界的质疑**：AI 基本是在我的选择范围内递进，没有主动质疑"这个范围是否太大"或"这个功能是否必要"。例如，在我说"记忆三种都需要"时，AI 没有追问"MVP 阶段是否可以先做一种"。

3. **技术选型缺少对比分析**：当我说"Python + DeepSeek"时，AI 没有追问"是否考虑过 Go/Rust 以获得更好的性能"或"是否考虑支持多 LLM 供应商"。虽然最终选型可能不变，但缺少对比会让我在 SPEC 中的"技术选型理由"部分缺乏论证深度。

4. **Mock 策略讨论较浅**：问题 24 关于 Mock LLM 策略的讨论只有一轮，没有深入追问"规则驱动 mock 的具体设计"——这其实是 harness 项目的核心判定标准之一。

### 3.3 如果重做 brainstorming，我会改变什么

1. 在每个重大决策后，主动问 AI："这个选择有什么风险或 trade-off 是我应该知道的？"
2. 在范围确认时，让 AI 给出一个"最小可行版本"建议，避免过度设计。
3. 在技术选型时，要求 AI 给出 2-3 个替代方案的对比。
